Wrap the single `generate_candidates` RPC call (`feed/index.ts:881-896`) in an expanding/tiered-radius loop (D-01..D-03, PERF-01 SC#1). Tiers are fractions of the client-requested radius `[0.25, 0.5, 1.0]`; the loop stops once the pool reaches POOL_TARGET (~100, half the 200-row `p_limit`); the final tier == the requested radius so worst-case (sparse/rural) == today's single-shot, never slower or wider. Purpose: dense-urban requests finish under `statement_timeout` at a narrow radius and bail early; sparse requests fall through to today's full-radius behavior. The mobile response contract is byte-identical — everything from `const pool = candidates ?? []` (`:903`) downstream is untouched.
