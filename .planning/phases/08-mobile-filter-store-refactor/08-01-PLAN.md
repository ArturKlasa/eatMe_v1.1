<slice_typing_decision>
Use plain factory functions for slices, NOT Zustand's `StateCreator` generic, because there is NO middleware in the chain (the store is `create<FilterState & FilterActions>((set, get) => …)` with a bare mutator tuple `[]`). Each slice file exports a factory of the form:

`export const createXxxSlice = (set, get) => ({ ...slice methods... })`

typed against the shared `FilterState & FilterActions` store type imported from `./types`. Type `set`/`get` using Zustand's parameter types for a `create<FilterState & FilterActions>` store (e.g. import the `StateCreator` param types from `zustand` for the `set`/`get` signatures only, or reuse `Parameters<...>` of the create callback) — do NOT wrap slices in `StateCreator<...>`. `index.ts` spreads all slice factories inside the single `create()` call so the final composed store type and the single `create()` call are unchanged. State this typing choice in the SUMMARY.

Why cleanest with no middleware: `StateCreator<FilterState & FilterActions, [], [], SliceShape>` adds mutator-tuple ceremony that buys nothing when the chain is empty; plain factories keep `set`/`get` identical to today's closure and avoid per-slice partial-type friction.
</slice_typing_decision>
